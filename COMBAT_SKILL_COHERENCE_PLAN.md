# Combat & Skill Coherence (Mythras-influenced)

Status tags follow the `ACTIVE_PROJECTS.md` convention: mark a phase `[COMPLETE]` in its heading below once it's implemented and verified in-game. Phases start `[NOT STARTED]`.

## Context

Follow-on to the completed `rpg-stats-overhaul` and `constitution-derived-stats` work. Those projects built the attribute layer (Str/Agi/Per/Con on `RpgStats`) and a unified `CombatResolver`, but left the attribute→bonus math incoherent: seven different formulas convert a 1-100 attribute into a combat effect, each with its own invented bucket size or scalar (`floor(per/15)*0.05` for crit, `floor(per/15)*0.02` for ranged aim, `floor(agi/25)*0.02` for melee aim, `floor(str/20)` capped for damage, `floor(con/20)` for sickness resist, `agi*0.5` for dodge, `con*0.2` for HP). Separately, `PlayerSkills` already gives melee/ranged their own kill-gated level (`meleeLvl`/`rangedLvl`, milestone `5*2^level`) feeding hit chance directly — but dodge has no equivalent skill track, it's pure `agi*0.5`.

This project's goal, arrived at via design discussion referencing Mythras (percentile skills, opposed rolls, characteristic-seeded skills) and D&D (flat modifiers, static AC) as reference points: make the existing skill/attribute split *coherent* — not build a comprehensive universal system. Scope is melee, ranged, and dodge/defense specifically.

## Overall summary — how the system will work

- **Attributes stay 1-100** (Str/Agi/Per/Con). They represent current physical state, and continue to fluctuate via `SurvivalCascade`'s continuous nutrition/hydration/energy multiplier. They are *not* discretized down to a D&D-style 1-20 — the needs system depends on fine-grained resolution to feel smooth rather than snapping across bucket boundaries.
- **Skills (Melee, Ranged, Defense) become percentages, not discrete levels.** Same 0-100ish numeric language as attributes, so the whole character sheet reads in one consistent unit. Each skill percentage feeds its combat roll directly (1 point = 1 percentage point, matching how `skillLvl` already works internally).
- **Skills are seeded once from a relevant attribute pair at creation/first use**, Mythras-style, then grow independently through use — not continuously re-derived from live attributes. Melee seeds from **Strength + Agility**, Ranged seeds from **Agility + Perception**, Defense seeds from **Agility + Perception**. Agility runs through all three (the universal combat-reflex stat); Strength is Melee-only; Perception is Ranged/Defense-only. This keeps two genuinely separate resources: attributes = physical conditioning (fast-moving, needs-coupled), skills = trained competency (slow-moving, use-coupled, immune to a bad hunger tick). Continuous derivation was considered and rejected — it would make skill training and attribute training redundant with each other and would leak needs-based fluctuation into "how good you are at fighting," which reads wrong.
- **Skill growth moves from kill-gated milestones to per-hit percentage growth, rescaled to keep roughly the same pace** (see Locked design decisions for the concrete `12.5 * 2^level` hits formula). A derived "Level" (`percent // 10` or similar) can still be surfaced in UI so the discrete level-up feeling isn't lost, even though the underlying number is continuous.
- **Dodge is generalized into a Defense skill that every entity capable of evading has** — but not every combat target. Turrets and structures stay defenseless on purpose (see below), so the fork narrows from three cases to two: entities that can evade (player/NPC/zombie, resolved via opposed roll) and inanimate/stationary targets (turrets/structures/doors/windows, resolved as a bare attacker's-hit-chance check, no defender roll at all). Combat resolves as an opposed roll for the first group: attacker's hit roll (skill + attribute bonus) vs. defender's Defense roll, better degree of success wins.
- **Defense is passive and free** — no AP cost, available on every incoming attack for every entity. An optional same-turn-only "Active Defense" (spend this turn's AP for a temporary bonus, instead of attacking/moving) can sit on top later, but there is no banked/borrowed-AP mechanic.
- **Defense is seeded from Agility + Perception** (reflexes + awareness). A successful *contested* defense (attacker's hit roll succeeded, defender then won the opposed roll) advances the Defense percentage and grants a split trickle of Agility+Perception XP — mirroring `MELEE_SKILL_UP`'s existing side-effect pattern, just sourced from both seed attributes instead of one. An attack that already missed on the attacker's own roll never triggers a defense check, so it grants nothing — this keeps Defense progress tied to actually-contested rolls, not mere presence in combat.
- **All attribute→bonus formulas are unified under one continuous (non-bucketed) conversion**, rather than each system inventing its own floor/divisor. Damage bonus and max-HP scaling can reasonably stay on their own separate curve (they aren't probability nudges), but every "attribute nudges a percentage" site — crit, aim, dodge/defense, sickness resist — shares one formula shape: `attrMod(attr) = (attr - 20) * STEP`, continuous (no `floor()`), pinned to a shared baseline of 20, with a per-site weight multiplier that's explicit and tunable rather than a hidden ad hoc bucket size.

## Locked design decisions (from design discussion — do not re-litigate)

- **Keep attributes at 1-100.** Rejected 1-20: it would make needs-based fluctuation chunkier and less responsive, undermining a mechanic that already works well.
- **Skills are seeded once, not continuously derived.** The attribute contributes a starting/floor value and a small ongoing secondary bonus (as Perception/Agility already do for aim); it does not recompute the skill's level on every tick.
- **Concrete seed formula, deliberately small relative to earned progression**: `seedLevel(attr1, attr2) = max(0, floor(((attr1-20) + (attr2-20)) / 10))`, reusing the same baseline-20 convention as `attrMod`. This means the player *does* effectively pre-level their combat skills through attribute investment at character creation — that's the direct, intended consequence of "skills are seeded from attributes," not an oversight. It reads narratively as natural aptitude (a strong, quick person is modestly better with a weapon before any training), not trained competency. The magnitude matters: because `12.5 * 2^level` hits per level is steep (level 5 costs ~400 hits, level 10 costs ~12,800), the seed must stay small — this formula caps a realistic creation-screen allocation to roughly a 0-6 level head start, nowhere near what earned play produces, so creation-point investment nudges the starting line without substituting for the grind.
- **Character creation already exists and already has the right UI pattern to extend** — `CharacterCreator.tsx` (80-point pool, each attribute floored at 10) already shows live derived-effect bullets per attribute (melee damage bonus, dodge %, crit %, Max HP, etc.) sourced from `CombatResolver`/`SurvivalCascade` formulas, mirrored by the in-game `PlayerSkillsUI.tsx`. Showing the new skill-seed level (e.g. "Melee starts at level 3 (3% bonus)") is an additional bullet in the same established pattern, not a new UI mechanism.
- **Skill growth should be reward-for-action, not reward-for-kill.** Landing a hit should count toward skill progress even without a kill.
- **Melee seeds from Strength + Agility; Ranged seeds from Agility + Perception.** Each successful hit grants a split trickle of XP to its skill's seed pair (Melee → Str+Agi, Ranged → Agi+Per), mirroring the Defense pattern below — this supersedes the old milestone-only `MELEE_SKILL_UP`/`RANGED_SKILL_UP` side effect (which granted Str/Per, and only fired at rare kill-gated level-ups) since skill progress itself is now per-hit, not per-milestone.
- **Melee/Ranged skill growth is per successful hit, rescaled from the old kill-milestone curve to keep pace roughly the same.** `getNextKillMilestone(level) = 5 * 2^level` kills becomes `12.5 * 2^level` hits (5 × the ~2.5 average hits-per-kill for a regular zombie across common weapons — Knife 4, Wooden Bat 3, Machete 2, Fire Axe 1, 9mm Pistol 2 hits-to-kill on a 10hp zombie). A kill's final blow is itself a landed hit, so it isn't double-counted on top. This keeps average pace close to today's for typical play; deliberately fighting with an underpowered weapon against a tough target will level faster in wall-clock terms than the old system allowed (more hits land per kill) — accepted as fine, not something to guard against.
- **Defense becomes a skill every entity capable of evading has** (player, NPC, zombie), resolved via opposed rolls. **Turrets and structures explicitly do not get a Defense** — they're the equivalent of attacking a door or window, a fixed target with no evasion, so they stay a bare attacker-hit-chance check with no defender roll. This isn't an oversight to fix later; it's the correct model for an inanimate target.
- **Zombies get a flat, per-type-tunable Defense value, not an attribute-derived one** (zombies have no `RpgStats`, so no `attrMod` applies and it never grows through use). The flat number lives on the zombie type definition, so different zombie archetypes can differ: a runner zombie gets a higher flat Defense (harder to hit), a fat/shambler zombie gets a lower one (easier to hit), a regular zombie sits at a tuned baseline. This replaces today's single `stumbleEvasion` value with a per-type stat rather than one constant for all zombies.
- **Attribute bonuses should share one formula shape**, not per-system bucket sizes. Damage/HP scaling (already continuous, e.g. `con*0.2`) are the model to extend to the others; the bucketed `floor(x/N)` formulas are the ones to replace.
- **Defense has no AP cost.** Considered banking AP specifically to enable defense, and rejected it: ending a turn holding AP in reserve "just in case" is never a real choice — if it's ever correct it's always correct, which means it isn't a choice. Passive, free Defense is the coherent answer, matching Mythras (Dodge/Parry isn't paid for, it just responds to being attacked). A same-turn "Active Defense" upgrade is a possible later addition (see Out of scope) but is not required for the base design.
- **Defense is seeded from Agility + Perception, both**, not Agility alone. XP for a successful defense splits between the two, gated on the roll being genuinely contested (attacker's hit succeeded, defender's opposed roll then won) — a defense against an attack that would have missed anyway grants nothing.
- **Unified bonus formula: `attrMod(attr) = (attr - BASELINE) * STEP`, continuous, no bucketing.** `BASELINE = 20` (matches the default value already used for Strength/Perception/Constitution). Per-site magnitude is an explicit weight on top of the same shared function, not a separately-invented divisor:

  | Site | Formula (STEP = 0.0015 provisional) | Attribute | Max (attr=100) | Min (attr=1) |
  |---|---|---|---|---|
  | Melee hit nudge | `(attrMod(str) + attrMod(agi)) / 2` | Str+Agi avg | +0.12 | −0.029 |
  | Ranged hit nudge | `(attrMod(agi) + attrMod(per)) / 2` | Agi+Per avg | +0.12 | −0.029 |
  | Defense bonus | `(attrMod(agi) + attrMod(per)) / 2` | Agi+Per avg | +0.12 | −0.029 |
  | Sickness resist fraction | `clamp(attrMod(con) * 5, 0, 0.6)` | Constitution | 0.60 (clamped) | 0 (clamped) |

  `STEP` and the sickness-resist ×5 weight are starting points for playtest tuning, not final — same "tune" caveat the Constitution project used for its own constants. The *shape* (continuous, baseline-relative, one shared function with visible per-site weights instead of seven ad hoc bucket sizes) is the locked part.
- **Follow-up flagged, not blocking**: Agility currently defaults to 40 (vs. 20 for Str/Per/Con), almost certainly tuned around the old `agi*0.5` dodge formula. Once `attrMod` lands, an average/untrained character reads as a flat 0 modifier only if Agility's default is retuned down to 20 to match the others — otherwise default characters start with a built-in positive melee-hit/defense nudge from Agility alone. Decide at implementation time.
- **Also bucketed, same problem, outside this table's scope**: `strengthDamageBonus = min(5, floor(str/20))` is a damage formula, not a probability nudge, so it wasn't unified above per the earlier "damage/HP stay on their own curve" decision — but it has the identical bucketing problem. Worth flattening to a continuous formula in the same pass even though it's not part of the shared `attrMod` table, since it's the same class of fix.
- **Melee weapons no longer carry a per-weapon accuracy stat.** Removed `weaponStats.hitChance` (0.65-0.90, varied per melee weapon) from the melee/NPC-melee hit formula entirely, replaced by one flat `BASE_MELEE_HIT_CHANCE = 0.75` (provisional) shared by every melee weapon. Reasoning: the per-weapon value stacked unclamped on top of skill+attribute, and per the actual itemization data it correlated *with* damage rather than trading off against it (Fire Axe was simultaneously the most accurate AND the hardest-hitting melee weapon, Knife the least accurate AND the weakest) — so it wasn't creating a meaningful choice, it was just another hidden "better weapon" stat that also quietly undercut the "trained skill is the dominant hit-chance lever" principle already locked in above. Weapon differentiation now lives in damage, crit, and (future) speed/AP cost — not accuracy. Ranged's per-weapon accuracy (distance falloff, `minAccuracy`, scope/laser range extension) is unaffected — that's modeling range/optics, a different and still-valid axis, not a flat weapon-handling accuracy stat.
- **Crit is a degree-of-success on the hit roll itself, not a separate roll.** Removes `weaponCritChance` + `perceptionCritBonus` entirely (hence no "Crit bonus" row in the `attrMod` table above — it's obsolete). The engine already rolls low-is-good (`gameRandom.next() <= hitChance`); a crit fires when the roll lands well under the threshold, not just under it: `roll <= hitChance / CRIT_DIVISOR` (provisional `CRIT_DIVISOR = 5`, mirroring Mythras' special-success band). Perception still matters for crit, but only *indirectly* — it's already folded into `hitChance` via the ranged/melee aim nudge, which widens the crit band along with the hit band. One roll now determines hit, miss, and crit together, instead of a hit roll plus an independent crit roll. This is a bigger structural change than the rest of the unification (touches `rollPlayerMelee`/`rollPlayerRanged`/`rollNpc`'s control flow, not just a formula swap) — sequence it as its own phase rather than folding it into the `attrMod` rollout.

## Implementation phases

Ordered by dependency: formula unification first (everything else reads from it), then attacker-side crit (touches only the roll functions Phase 1 already stabilized), then the melee/ranged growth-model swap, then the structural Defense/opposed-roll rewrite (isolated on its own since it's the highest-risk phase, mirroring how the Constitution project isolated its `recalcCharacter` phase), then character-creation seeding last since it depends on the skill fields Phases 3-4 create.

### Phase 1 — Unify attribute-bonus formulas (`attrMod`) [COMPLETE]

Pure formula swap inside `CombatResolver.js`, no data-model changes, no new fields anywhere.

1. Add `attrMod(attr, step = STEP)` as a module-level helper (`BASELINE = 20`, `STEP = 0.0015` as named, tagged-provisional constants).
2. Replace `meleeAimBonus`/`perceptionAimBonus` (~line 54-60) with the averaged two-attribute versions from the locked table: melee = `(attrMod(str) + attrMod(agi)) / 2`, ranged = `(attrMod(agi) + attrMod(per)) / 2`.
3. Replace `sicknessResistLevel`/`sicknessResistFraction` (~line 68-75) with `clamp(attrMod(con) * 5, 0, 0.6)`.
4. Flatten `strengthDamageBonus` (~line 63-65) from `min(5, floor(str/20))` to a continuous form, e.g. `Math.min(5, Math.max(0, (str - 20) * 0.05))` (provisional, same tune-later caveat as the rest).
5. Remove `perceptionCritBonus` only (leave `weaponCritChance` in place for now) — crit becomes slightly less feature-rich for one phase (no perception bonus, base weapon chance only) until Phase 2 replaces the whole crit mechanism. Not a regression worth blocking on, just a known transitional state.
6. Retune Agility's default from 40 → 20 in `RpgStats.js` constructor, `Entity.js`'s `defineAccessors` block, and `EntityFactory.js` (`createPlayer`/`createNPC`) — bundled here because it's directly downstream of `attrMod`'s baseline-20 assumption.

**Verify**: `npm run check`; spot-check hit/dodge/sickness-resist numbers for a default (baseline-attribute) character are ~0 modifier as intended; confirm crit still fires via base weapon chance alone in this interim state.

### Phase 2 — Crit as degree-of-success [COMPLETE]

Touches attacker-side roll control flow only (`rollPlayerMelee`, `rollPlayerRanged`, `rollNpc`), independent of the Defense rework in Phase 4.

1. Remove `weaponCritChance` and its separate roll entirely.
2. In each attacker roll function, capture the single `roll = gameRandom.next()` already used for the hit check; after confirming `roll <= hitChance`, set crit when `roll <= hitChance / CRIT_DIVISOR` (provisional `CRIT_DIVISOR = 5`) instead of rolling again.
3. Damage-on-crit (`max * 1.5`) stays as-is, just gated by the new condition.
4. Decide at implementation time whether any per-weapon crit flavor is worth preserving (e.g. a weapon-specific divisor tweak) or whether losing weapon crit differentiation entirely is acceptable — flagged as a possible balance loss to watch in playtest, not pre-decided.
   - **Decided**: dropped entirely. `weaponCritChance` (item-level `critChance` override) is deleted along with its only call sites — crit is now purely margin-of-roll, no per-item override. Turret crit (`rollTurret`, its own separate `0.05 + (rangedLvl-1)*0.05` roll) is intentionally untouched — Phase 2 scope was the three attribute-driven roll functions only, and turrets have no Perception/attributes to fold into a shared roll anyway.

**Verify**: crit rate roughly matches the expected band for a given hit chance (e.g. ~60% hit chance → crit on ~12% of all attempts, ~20% of successful hits); `npm run check`.

3. On milestone crossing, change the `AttributeProgressionManager.recordAction` call from single-attribute (`MELEE_SKILL_UP`→Str/Per today) to the split pattern: Melee → Str+Agi, Ranged → Agi+Per. New action names `MELEE_HIT`/`RANGED_HIT` replace `MELEE_SKILL_UP`/`RANGED_SKILL_UP` entirely — the old actions granted a huge lump ("enough XP to guarantee a stat roll") sized for rare kill-gated milestones; the new ones grant a flat 4 XP per attribute per hit (provisional, tune later), sized like other frequent per-action grants (`DODGE_SUCCESS`=15, `HEARING_SUCCESS`=5).
